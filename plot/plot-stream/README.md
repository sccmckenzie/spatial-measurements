# plot-stream

a small, somewhat clunky web server that gives you near real-time visualization of
wafer maps as they're generated by the `scanner` java app.

use at your own risk — there are some issues with runaway browser memory that still
need to be addressed (see [known issues](#known-issues)).

## what it does

- `server.py` — a FastAPI app that serves a static page and streams measurements over
  [Server-Sent Events](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events).
- `static/timeline.js` — a Plot timeline of scan activity over time.
- `static/wafer.js` — a faceted grid of wafer maps, colored by `measurement_value`.

data flows one way: `scanner` writes rows into `raw.measurement` in Postgres, the server
polls that table, and the browser re-renders on each push.

## running it

the server reads from Postgres directly, so you need the database up and the `scanner`
app writing to it. connection details are hard-coded near the top of `server.py`:

```
DATABASE_URL = 'postgresql://scanner_ro:applesauce@localhost:5432/postgres'
```

Plot and d3 are vendored one level up in `../vendor`, so there's no build step and no CDN
dependency — the server mounts that directory at `/vendor`.

```
uv run uvicorn server:app --reload
```

then open http://localhost:8000.

## technical notes

- **transport.** the browser opens an `EventSource` to `/stream` and the server holds the
  connection open, pushing a JSON payload on each tick. SSE (rather than websockets) is a
  deliberate choice: the data only flows server → client, and `EventSource` reconnects
  on its own if the stream drops.
- **polling, not notifications.** `/stream` re-queries `raw.measurement` every
  `POLL_INTERVAL_S` (0.1s) and ships the full result each time — there's no diffing or
  cursor. the query is capped to the first `SCAN_LIMIT` (10) scans by `scan_id`. this is
  simple and stateless but wasteful; the intended fix is Postgres `LISTEN`/`NOTIFY` on
  inserts (see the `TODO(v2)` in `server.py`).
- **full re-render each tick.** both `timeline.js` and `wafer.js` throw away the previous
  chart and rebuild a fresh Plot figure from the whole payload on every message. no
  incremental updates.
- **two independent streams.** `timeline.js` and `wafer.js` each open their *own*
  `EventSource`, so the server runs the poll query twice per tick and every client costs
  two long-lived connections. consolidating to a single shared stream would roughly halve
  the DB load.
- **wafer layout.** `wafer.js` lays scans out in a `COLS × ROWS` (5 × 2) facet grid and
  sizes cell bands from the observed coordinate extent, so the maps stay square as scans
  fill in.

## known issues

- **runaway browser memory.** at 10 pushes/second across two streams, the tab allocates
  and discards a lot of SVG very quickly. detached Plot figures don't always get collected
  promptly, so memory climbs the longer the page stays open. lowering `POLL_INTERVAL_S`,
  collapsing to one stream, and moving to `LISTEN`/`NOTIFY` would all help.
- **hard-coded config.** the database URL, poll interval, and scan limit are all constants
  in `server.py`. fine for a demo, less so for anything else.
